<h1>

  Perspectivas sobre como ser um arquiteto de software eficaz

 </h1>

<p> Arquitetura de software é trabalho social. Não é raro o arquiteto de software também ser o chefe do projeto e também ser um CTO.</p><br> </br>

 

<a>Como arquiteto, devemos ser capazes de responder a três perguntas: o que? quão? e com o que?</a>

 

  <p>Um arquiteto de software deve entender qual é o requisito. Então, como sabemos disso? Conversando, conversamos com o proprietário do produto / que solicita o projeto e conversamos com todas as partes, incluindo a equipe.</p><br> </br>

 

<p>Em seguida, o arquiteto de software deve avaliar os recursos. Por exemplo, o objetivo é construir um Facebook em um ano, mas temos 3 desenvolvedores juniores. Então, é possível? Não, sem recursos extras, então vamos conversar com o pessoal de RH se pudermos contratar novos desenvolvedores ou, digamos, se podemos aumentar o tempo ou diminuir os requisitos.</p>

 

<p>Sobre os recursos, digamos que é uma startup e a empresa não escolheu uma tecnologia, portanto, devemos escolher uma tecnologia com base nos recursos (desenvolvedores, licenças, servidor, se houver) e começar a marcar um roteiro. Por exemplo, se a equipe está alinhada com o Visual Basic 6 e é suficiente trabalhar, vá em frente e iremos desenvolver com o VB6. No entanto, se somos uma startup (não há tecnologia ou equipe), poderíamos escolher a melhor tecnologia para o trabalho.<p>

 

</p>

 

<p>Por fim, criamos o modelo de banco de dados, a maquete da tela, o caso de uso (se houver), a UML (geralmente são lixo inútil, mas algumas pessoas adoram) e tal. Então, podemos começar? NÃO. Devemos conversar (novamente) com o proprietário do produto, mostrar a maquete, o caso de uso (ou fluxos de trabalho, se houver) e o proprietário do produto pode solicitar alterações.<p>

 

</p>

 

 

Os arquitetos de software geralmente interagem com os clientes e as partes interessadas de um projeto. Eles - idealmente - precisarão se envolver para descobrir exatamente quais tipos de requisitos o cliente está procurando, sendo capazes de garantir que os requisitos sejam requisitos de negócios e não apenas "fazer com que esta página tenha uma barra de menu superior flutuante".

 

Total aparte: Sim, eu estive em um projeto em que literalmente passamos horas de reuniões com CEOs de duas grandes companhias de seguros apenas para decidir se usaríamos um menu superior flutuante no site 😜. Acontece...

 

De qualquer forma, o arquiteto precisa garantir que a direção técnica do projeto seja viável / viável, atendendo às necessidades do negócio.
